# Requirements v2 (Migration Reset)

Last updated: 2026-04-15
Project: `openclaw-ai-marketing-note-company`

## 1. この再定義の目的

PC移行を踏まえて、本プロジェクトの要件を整理し直す。
今回の見直しでは、"GitHubをつなぐこと"自体を目的にせず、**なぜGitHub連携が必要なのか**を先に明確化する。

## 2. 先に結論

本来の目的は、**OpenClaw上で動かす AI / Webマーケ運用体制を、1台のPCや一時的なローカル状態に依存させず、継続運用できる形にすること**。

つまり GitHub 連携の役割は、単なる push 先ではなく、以下を満たすための基盤である。

- プロジェクトの最新版を一元管理する
- PCを移しても作業を再開できる
- 要件定義、プロンプト、runbook、テンプレ、ログ方針を失わない
- シエルが今後の改善を積み上げられる状態を作る

## 3. GitHub連携の本来の目的

### 3.1 主目的
- プロジェクトの**正本 (single source of truth)** を持つこと
- PC移行や環境差分があっても、作業再開点を失わないこと
- 運用設計、プロンプト、テンプレ、設定方針を継続的に更新できること

### 3.2 副目的
- 変更履歴を残す
- どこまで整備済みかを見える化する
- 今後、別PCや別環境でも再利用しやすくする
- 将来の自動化や運用拡張の土台にする

### 3.3 今回わかったこと
- Browser Relay や Chrome 連携は、GitHub同期の**手段のひとつ**でしかない
- それが不安定でも、プロジェクトの本来目的は消えない
- したがって、**GitHub連携の要件は「ブラウザ連携成功」ではなく「安定して同期・復旧できること」** に置き直すべき

## 4. プロジェクトの上位目的

本プロジェクトは、参照したX投稿の“AI運用組織”の構造を参考にしつつ、**Claude Codeそのものではなく OpenClaw 上で同等の運用構造を再現する**ことを目的とする。

対象テーマは以下。
- AI
- Webマーケティング
- 広告運用

基本導線は以下。
- X
- 無料note
- 有料note

初期運用方針は以下。
- 自動投稿なし
- AI下書き + 人確認 + 手動公開
- 5人部隊構造をできるだけ維持

## 5. 今回の再定義で追加された重要要件

### 5.1 継続運用要件
- PCが変わっても再開できること
- 現在の構成や進捗がファイルとして残っていること
- ローカルだけに閉じた設定に依存しすぎないこと

### 5.2 再現性要件
- 新PCでセットアップ手順を辿れば再開できること
- GitHubからプロジェクト一式を取得できること
- ローカル固有の秘密情報は分離されていること

### 5.3 同期要件
- GitHub同期は安定手段を優先すること
- Browser Relay は必須条件にしないこと
- SSH / PAT / GitHub CLI / ブラウザ連携のうち、最も安定する方式を採用できること

### 5.4 運用資産管理要件
GitHubで最低限管理すべき対象は以下。
- 要件定義
- KPI案
- prompts
- runbooks
- workflows
- samples
- ログ雛形
- ブランド設定のテンプレ

## 6. GitHub連携の成功条件

GitHub連携が成功したとみなす条件を、以下に置き直す。

### 必須
- リポジトリにプロジェクト土台が保存されている
- 新PCで clone / pull により再取得できる
- 変更を commit / push できる
- 最新状態がローカルだけに閉じない

### できれば欲しい
- シエル側から差分確認しやすい
- 将来の複数端末運用に耐える
- GitHub認証がブラウザ依存すぎない

### 必須ではない
- Browser Relay の常時安定運用
- Chrome拡張の利用
- ブラウザ越しの手動補修前提フロー

## 7. 非機能要件の見直し

### 7.1 安定性
- 一時的なブラウザ不調で全体が止まらないこと
- PC移行後でも30〜60分程度で復旧方針を立てられること

### 7.2 可搬性
- WSL2 / Windows / 他PCでの再開を想定する
- 依存関係や接続方式が文書化されていること

### 7.3 安全性
- 認証情報をリポジトリに直書きしない
- ブランド未確定情報や運用上の秘密は適切に分離する

## 8. 現時点の判断

現時点では、以下の理解に更新する。

- GitHub連携の本当の役割は**作業継続性の担保**
- Browser Relay はそのための補助手段
- 今回のPC移行で重要になったのは、**環境が変わってもプロジェクトを再開できる設計**
- したがって次の要件整理は、GitHub認証方式だけでなく、**再開性・再現性・同期の安定性** を中心に進めるべき

## 9. 次に詰めるべき論点

1. GitHub同期の第一手段を何にするか
   - SSH
   - PAT

   - Browser Relay は補助扱い

2. 新PCセットアップ手順をどこまで文書化するか

3. GitHub管理対象とローカル専用情報をどう分けるか

4. 今後、OpenClawを Windows ネイティブ / WSL2 のどちらで主運用するか

## 10. この時点での提案

次フェーズでは、以下の順で進めるのが良い。

1. 本 v2 をベースに認識合わせ
2. GitHub連携の目的を「安定同期」に固定
3. 認証方式をブラウザ依存から外す案を優先検討
4. 新PC再開手順の runbook を作る
5. その後に実運用ファイル整備へ戻る
